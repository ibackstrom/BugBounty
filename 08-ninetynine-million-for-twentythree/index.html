---[  bugbounty notes, vol 1, article 08

-------------------------[  ninety nine million euros of insurance for twenty three

--------[  r1gor

second hand marketplace. shipping app. you print a label, pay a few extra euros, parcel is insured up to 500 EUR if it goes missing. nice little side service.

three calls to buy a label.

POST /api/orders/create -> get order uuid PUT /api/orders/summary -> get a price quote POST /api/orders/purchase -> get a payment link

the summary call is where the price is decided. you tell the server what the parcel is worth, the server tells you what coverage costs.

----[ the call

PUT /api/orders/summary/<uuid>
{
  "rate_uuid": "...",
  "content_value": 10,
  "coverage_applied": true
}

content_value is the value YOU declare. server trusts it. server uses it to price the coverage. you can guess where this goes.

i sent a bigger number.

{ "content_value": 99999999, "coverage_applied": true }

response, http 200:

{
  "coverage_price":          { "amount": "20.00", "currency": "EUR" },
  "label_price":             { "amount": "2.99",  "currency": "EUR" },
  "max_coverage_price":      { "amount": "500",   "currency": "EUR" },
  "min_content_value_price": { "amount": "25",    "currency": "EUR" },
  "total_price":             { "amount": "22.99", "currency": "EUR" },
  "extra_coverage_price":    { "amount": "20.00", "currency": "EUR" }
}

read it twice. same json object. server says max_coverage_price is 500 EUR. then prices ninety nine million worth of coverage at 20 EUR flat. total bill twenty two euros and ninety nine cents.

the rule and the violation are in the same response. like a speed limit sign painted on the side of a getaway car.

----[ bonus rounds

content_value zero, coverage on:

{ "coverage_price": "0.00", "total_price": "2.99", ... }

free coverage. attached as a real line item to the order. min content value is twenty five EUR according to the same response. enforced nowhere.

parcel size, second bug, same shape. order created as size s. purchase call accepts parcel_size as a top level field. send hl. server prints the bigger label and charges the small label price. price was computed once at create time and never recomputed.

----[ why it works

three bugs, one parent. server trusts the client to declare the value, the size and whether the rule applies on this particular request. the fields max_coverage_price and min_content_value_price exist in the response. somebody wrote them. they just are not wired to the validator.

fix is the boring one. recompute on the server at the moment of purchase, from the order row, not from the request body. validate the declared value against the maximum that the server itself just returned. pick the parcel size from persisted state.

----[ retest

reported. came back a week later. summary still misquotes ninety nine million as a twenty euro line item. purchase endpoint now returns 422 with an empty errors array for any payload, including the boring baseline that worked before. somebody plugged the exit door and left the wrong price tag on the wall.

partial fix. root cause unfixed. reportable as medium today, would be high again the moment the purchase guard regresses.

----[ takeaway

when a server returns its own rule next to its own decision, read both. the response is sometimes confessing.
